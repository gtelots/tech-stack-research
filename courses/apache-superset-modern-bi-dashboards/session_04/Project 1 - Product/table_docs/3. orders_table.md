# Orders Table Documentation

## Table Overview
The `orders` table is the central transaction record in our business intelligence database. It captures detailed information about every purchase, including products bought, pricing, discounts, shipping, and customer information. This comprehensive table enables deep analysis of sales patterns, customer behavior, and business performance across multiple dimensions.

## Table Structure

| Column Name           | Data Type      | Constraints        | Description                                           |
|----------------------|----------------|-------------------|-------------------------------------------------------|
| order_id              | INTEGER        | PRIMARY KEY       | Unique identifier for each order                       |
| customer_id           | INTEGER        | FOREIGN KEY       | Reference to customer who placed the order             |
| product_id            | INTEGER        | FOREIGN KEY       | Reference to product purchased                         |
| quantity              | INTEGER        | NOT NULL          | Number of items ordered                                |
| unit_price            | DECIMAL(10,2)  | NOT NULL          | Price per item at time of purchase                     |
| total_amount          | DECIMAL(10,2)  | NOT NULL          | Total order amount including discounts, tax, shipping  |
| discount_percentage   | DECIMAL(5,2)   | DEFAULT 0.00      | Percentage discount applied to the order               |
| discount_amount       | DECIMAL(10,2)  | DEFAULT 0.00      | Monetary amount of discount                            |
| tax_amount            | DECIMAL(10,2)  | DEFAULT 0.00      | Tax applied to the order                               |
| order_date            | TIMESTAMP      | NOT NULL          | When the order was placed                              |
| status                | VARCHAR(20)    | NOT NULL          | Current order status                                   |
| payment_method        | VARCHAR(50)    | -                 | Method used for payment                                |
| shipping_method       | VARCHAR(50)    | -                 | Method used for shipping                               |
| shipping_cost         | DECIMAL(8,2)   | DEFAULT 0.00      | Cost of shipping                                       |
| is_gift               | BOOLEAN        | DEFAULT FALSE     | Whether the order is a gift                            |
| is_returned           | BOOLEAN        | DEFAULT FALSE     | Whether the order was returned                         |
| return_date           | TIMESTAMP      | -                 | When the order was returned (if applicable)            |
| return_reason         | VARCHAR(200)   | -                 | Reason for return (if applicable)                      |
| delivery_date         | TIMESTAMP      | -                 | When the order was delivered                           |
| sales_channel         | VARCHAR(50)    | -                 | Channel through which the sale was made                |
| created_at            | TIMESTAMP      | DEFAULT CURRENT_TIMESTAMP | When the order record was created             |
| updated_at            | TIMESTAMP      | DEFAULT CURRENT_TIMESTAMP | When the order record was last updated        |

## Indexes
- **Primary Key**: `order_id` (automatically indexed)
- **Secondary Indexes**: 
  - `idx_orders_customer_id` - Improves performance when querying orders by customer
  - `idx_orders_product_id` - Improves performance when querying by product
  - `idx_orders_order_date` - Enables efficient time-based analysis

## Column Details

### order_id
- **Purpose**: Serves as the unique identifier for each order transaction
- **Implementation**: Auto-incrementing integer (SERIAL) that ensures uniqueness
- **Business Use**: Used as the reference number for order tracking and customer service
- **Range**: Positive integers starting from 1

### customer_id
- **Purpose**: Links each order to a specific customer record
- **Implementation**: Integer that references the customer_id in the customers table
- **Business Use**: Enables customer purchase history analysis and personalization
- **Constraints**: Foreign key constraint ensures data integrity with customer records

### product_id
- **Purpose**: Links each order to a specific product
- **Implementation**: Integer that references the product_id in the products table
- **Business Use**: Enables product sales analysis and inventory management
- **Constraints**: Foreign key constraint ensures data integrity with product records

### quantity
- **Purpose**: Records how many units of the product were purchased
- **Implementation**: Positive integer representing count
- **Business Use**: Used for inventory management, popular product analysis, and order fulfillment
- **Range**: Typically 1-5 in our sample data
- **Constraints**: Cannot be null (NOT NULL)

### unit_price
- **Purpose**: Stores the price of a single item at the time of purchase
- **Implementation**: Decimal number with up to 10 digits total, 2 of which are decimal places
- **Business Use**: Used for historical pricing analysis, as prices may change over time
- **Range**: Typically between 5.99 and 999.99 in our sample data
- **Constraints**: Cannot be null (NOT NULL)

### total_amount
- **Purpose**: Stores the final amount charged, including all adjustments
- **Implementation**: Decimal number with up to 10 digits total, 2 of which are decimal places
- **Business Use**: Used for revenue analysis, financial reporting, and order value analysis
- **Calculation**: (unit_price × quantity) - discount_amount + tax_amount + shipping_cost
- **Constraints**: Cannot be null (NOT NULL)

### discount_percentage
- **Purpose**: Records the percentage discount applied to the order
- **Implementation**: Decimal number with up to 5 digits total, 2 of which are decimal places
- **Business Use**: Used for discount strategy analysis and promotional effectiveness
- **Range**: Typically between 0 and 30 percent
- **Default**: 0.00 (no discount)

### discount_amount
- **Purpose**: Records the monetary value of discounts applied
- **Implementation**: Decimal number with up to 10 digits total, 2 of which are decimal places
- **Business Use**: Used for financial reporting and discount impact analysis
- **Calculation**: (discount_percentage / 100) × (unit_price × quantity)
- **Default**: 0.00 (no discount)

### tax_amount
- **Purpose**: Records the monetary value of tax applied to the order
- **Implementation**: Decimal number with up to 10 digits total, 2 of which are decimal places
- **Business Use**: Used for tax reporting, financial analysis, and regional tax impact studies
- **Calculation**: Typically calculated as a percentage of the pre-tax subtotal
- **Default**: 0.00 (no tax)

### order_date
- **Purpose**: Records when the order was placed
- **Implementation**: Timestamp with date and time information
- **Business Use**: Used for time-based sales analysis, seasonality studies, and trend identification
- **Constraints**: Cannot be null (NOT NULL)

### status
- **Purpose**: Indicates the current state of the order in its lifecycle
- **Implementation**: Variable character field with a maximum length of 20 characters
- **Business Use**: Used for fulfillment tracking, order status reporting, and process optimization
- **Example Values**: "Completed", "Processing", "Shipped", "Cancelled", "Returned"
- **Constraints**: Cannot be null (NOT NULL)

### payment_method
- **Purpose**: Records how the customer paid for the order
- **Implementation**: Variable character field with a maximum length of 50 characters
- **Business Use**: Used for payment trend analysis and financial reconciliation
- **Example Values**: "Credit Card", "PayPal", "Bank Transfer", "Cash on Delivery", "Gift Card"

### shipping_method
- **Purpose**: Records how the order was shipped to the customer
- **Implementation**: Variable character field with a maximum length of 50 characters
- **Business Use**: Used for logistics analysis, shipping cost optimization, and delivery performance
- **Example Values**: "Standard", "Express", "Next Day", "International", "Store Pickup"

### shipping_cost
- **Purpose**: Records the amount charged for shipping
- **Implementation**: Decimal number with up to 8 digits total, 2 of which are decimal places
- **Business Use**: Used for shipping margin analysis, delivery cost tracking, and pricing strategy
- **Default**: 0.00 (free shipping)

### is_gift
- **Purpose**: Indicates whether the order was marked as a gift
- **Implementation**: Boolean flag (true/false)
- **Business Use**: Used for gift-buying pattern analysis, seasonal gift trends, and special handling
- **Default**: FALSE

### is_returned
- **Purpose**: Indicates whether the order was returned by the customer
- **Implementation**: Boolean flag (true/false)
- **Business Use**: Used for return rate analysis, product quality issues, and customer satisfaction metrics
- **Default**: FALSE

### return_date
- **Purpose**: Records when a returned order was processed
- **Implementation**: Timestamp with date and time information
- **Business Use**: Used for return processing efficiency, time-to-return analysis, and seasonal return patterns
- **Constraints**: Only populated if is_returned is TRUE

### return_reason
- **Purpose**: Stores the customer's explanation for returning the order
- **Implementation**: Variable character field with a maximum length of 200 characters
- **Business Use**: Used for product quality improvement, return reason analysis, and customer satisfaction
- **Example Values**: "Damaged", "Wrong Size", "Wrong Item", "Not as Described", "Changed Mind"
- **Constraints**: Only populated if is_returned is TRUE

### delivery_date
- **Purpose**: Records when the order was delivered to the customer
- **Implementation**: Timestamp with date and time information
- **Business Use**: Used for delivery performance analysis, shipping time calculation, and customer experience metrics
- **Constraints**: Only populated for orders with status "Completed" or "Returned"

### sales_channel
- **Purpose**: Records which platform or method was used to make the purchase
- **Implementation**: Variable character field with a maximum length of 50 characters
- **Business Use**: Used for channel effectiveness analysis, marketing attribution, and platform investment decisions
- **Example Values**: "Website", "Mobile App", "In-store", "Phone", "Marketplace"

### created_at
- **Purpose**: Records when the order record was first created in the system
- **Implementation**: Timestamp that defaults to the current time when record is created
- **Business Use**: Used for system performance analysis, order processing time, and auditing

### updated_at
- **Purpose**: Records when the order information was last modified
- **Implementation**: Timestamp that automatically updates when the record is modified
- **Business Use**: Used for tracking order status changes, update frequency, and auditing

## Business Intelligence Applications

The orders table is the cornerstone of sales analysis and enables numerous business intelligence applications:

1. **Revenue Analysis**: Tracking total sales over time, by product, category, or customer segment.

2. **Discount Effectiveness**: Analyzing how different discount levels affect purchase behavior and overall margins.

3. **Shipping Analytics**: Evaluating shipping methods, costs, and delivery performance.

4. **Return Rate Analysis**: Identifying products with high return rates and common return reasons.

5. **Sales Channel Performance**: Comparing effectiveness and profitability of different sales platforms.

6. **Payment Method Trends**: Analyzing customer payment preferences and their impact on purchase behavior.

7. **Order Status Tracking**: Monitoring the efficiency of order fulfillment processes.

8. **Gift Purchase Analysis**: Understanding gift-buying patterns and seasonality.

9. **Tax Analysis**: Reviewing tax collection by region and purchase type.

10. **Customer Order Patterns**: When joined with the customers table, analyzing repeat purchase behavior and customer lifetime value.

## Data Generation
In our sample database, 1,000,000 synthetic order records are generated with realistic values across a two-year period. The data includes various order statuses, payment methods, shipping options, and return scenarios to enable comprehensive business analysis.
